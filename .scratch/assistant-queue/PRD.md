# PRD: 비서 능력 경계 + 캘린더 작업 큐 상태머신 + 폰 지침 재작성

## 목표

김비서를 **"안 된다고 말하지 않는" 비서**로 만든다. 폰 Gemini는 할 수 있는 건 즉시,
못 하는 건 **조용히 `{비서코드}-calendar` 작업 큐에 등록**하고, kim-1 클로드 코드 **워커**가
큐를 드레인하며 **상태머신**으로 처리한다. 폰 지침은 특정 비서코드를 하드코딩하지 않는
**제네릭** 지침으로 처음부터 다시 쓴다. (설계 결정: ADR-0004, 용어: `CONTEXT.md`.)

## 배경 원칙

- **프로젝트 = 비서 1명(kim-1)**. 멀티 비서는 프로젝트를 여러 개 만든다. 단, **폰 지침은 제네릭**(여러 비서코드를 넘나듦).
- 폰 Gemini는 지식 노드를 직접 고치지 않는다(그건 워커). vault는 여전히 worker-only.

## 요구사항

### R1 — 능력 경계 (직접 vs 큐)
- **직접(폰 즉시)**: 지메일 읽기 / 캘린더 일정 CRUD / Drive 읽기·생성(신규) / 지식 그래프 조회·순회(recall).
- **큐로(조용히 등록)**: 메일 발송·답장 / 기존 문서 수정 / 지식 저장소 저장(증류) / 길게 조사 / 그 외 전부.
- 폰은 큐행이어도 **사용자에게 "안돼"라 하지 않고**, "큐에 넣었다"고만 알린다.

### R2 — 작업(Task) 표현 (ADR-0004)
- 1 작업 = `{비서코드}-calendar` 이벤트 1개.
- 제목 `{아이콘}[{상태}] {요약}` — 상태 토큰은 기계 파싱용.
- 설명 3블록: `요청:` / `필요정보:` / `이력:`. **append-only·비파괴**.

### R3 — 상태머신
- `대기 → 작업중 → (성공 | 실패 | 정보필요)`.
- `정보필요`: **무슨 정보가 왜** 필요한지 명시. 사용자가 정보 붙이고 `대기`로 되돌리면 재개.
- `실패`: 자동 재시도 없음(사람이 판단). `성공`/`실패`도 캘린더에 **남김**(이력).
- 모든 전이는 `이력:`에 타임스탬프와 함께 기록.

### R4 — 시각 게이트 & 반복
- 워커는 **start ≤ 지금**인 `대기`만 집음(미래=대기). 이 시각이 스케줄.
- 반복 작업: 완료 시 **다음 시각으로 새 `대기` 작업 재등록**(설명 `반복:` 규칙 기반, 예 `평일 09:00`). 네이티브 RRULE 아님.
- 예: "평일 9시 해커뉴스 핫한 지식 정리해 나에게 메일" → 매 완료 시 다음 평일 9시로 재등록.

### R5 — 워커 (kim-1 클로드 코드)
- 큐 드레인 루프: 만기 `대기` 조회 → `작업중` → **작업 종류로 디스패치** → `성공/실패/정보필요` + 이력 + (반복이면) 재등록.
- 작업 종류 → 동작 매핑: 메일발송/답장(gws), 문서수정(gws), 지식저장(kim-learn→kim-sync), 조사(리서치), ….
- **자동 = 스케줄 루프**(로컬 `/loop` 또는 `schedule` cron). 결과는 작업 `이력:`에 append + 부수효과(메일 발송 등) 실제 수행.

### R6 — 폰 Gemini 지침 재작성 (제네릭, 처음부터)
- 비서코드 하드코딩 제거. "**비서 {코드}**"로 활성화, 세션 유지, 안 대면 되묻기, "비서코드 뭐야?"→활성 코드.
- **파일명 구조 상세 설명**: `{비서코드}_{no}_{namespace}_{doctype}_{visibility}_{title}` — 각 슬롯 의미, 폴더 무시·이름 타깃팅.
- **그래프 탐색 최대화**: 단일 노드로 끝내지 말고 엣지를 적극 순회해 서브그래프로 답.
- **Drive 조회 거부 방지**: "너는 Drive를 조회할 수 있다. 못 한다고 하기 전에 반드시 먼저 시도하라."
- **never-refuse → 큐**: 직접 못 하는 요청은 거부 대신 작업 큐 등록.

### R7 — README (비개발자용) + 예시 ~100개
- `jhs512/eb` README 많이 참고, **일반인 대상**(개발 용어 최소).
- **예시 ~100개**: 자연어 발화 → 김비서 동작/답 (직접·큐·상태조회·정보보충·반복 등 골고루).
  - 예: "김비서, 지금 비서코드 뭐야?" → "kim-1 입니다."
  - "지금 작업중인 일 뭐야?" / "정보필요로 멈춘 일 뭐 있어? 내가 정보 줄게"
  - "이건 바로 조회 말고 길게 조사해줘" → 큐 등록 후 결과를 작업에 보고.

## Non-goals
- 미디어(유튜브/음성) 전사, 벡터/의미검색은 이번 범위 밖.
- 폰이 큐 작업을 직접 처리하는 것(불가 — 워커 전용).
- 멀티 비서를 한 프로젝트에서 관리(멀티=프로젝트 여러 개).

## 성공 기준
- 폰 지침(제네릭)이 배포되어, "메일 보내줘/문서 고쳐줘/저장해줘/길게 조사해줘"에 **거부 없이 큐 등록**.
- 워커가 만기 `대기` 작업을 상태머신으로 처리하고 이력을 남김. `정보필요→대기` 재개 동작.
- 반복 작업이 완료 시 다음 시각으로 재등록.
- 상태머신·작업 파싱/렌더 순수 로직 **TDD 통과**.
- README(비개발자용) + 예시 ~100개.

## 테스트 전략 (TDD)
- 순수 모듈(작업 파싱: 이벤트↔{상태,요청,필요정보,이력}, 전이 검증, 렌더, 만기/반복 계산)을 `node:test`로 TDD.
- gws I/O(캘린더 read/patch/insert)·워커 디스패치는 얇은 셸 + 수동 실증(캘린더에 작업 1개를 대기→작업중→성공까지).

## 이슈 분할 (예정)
1. 작업 상태머신 순수 모듈(파싱/전이/렌더) — TDD
2. 시각 게이트 + 반복 재등록 계산 — TDD
3. 캘린더 작업 큐 I/O 셸(kim-cal 확장: due-list/transition/append-history/reenqueue)
4. 워커 루프 + 작업종류 디스패치(kim-work 스킬)
5. 폰 Gemini 지침 재작성(제네릭)
6. README(비개발자용) + 예시 ~100개
7. 리팩토링(기존 kim-cal 흡수/정리)
