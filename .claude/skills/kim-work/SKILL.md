# kim-work — 작업 큐 워커 (상태머신)

폰 Gemini가 직접 못 해서 `{비서코드}-calendar`에 쌓아둔 **작업**을 kim1(클로드 코드)이
드레인한다. 결정적 큐 I/O는 `scripts/kim-cal.sh`(→ 테스트된 task/schedule)가, 실제 일과
판단은 이 스킬이. 상태·용어는 `CONTEXT.md`·ADR-0004.

## 루프

1. **만기 대기 조회**
   ```bash
   bash scripts/kim-cal.sh due     # eventId <TAB> 제목 (start ≤ 지금 & 상태=대기만)
   ```
   비면 종료. 미래 시각 작업은 자동 제외(시각 게이트).

2. **작업마다 (하나씩)**:
   1. `set <id> 작업중 "착수"` — 상태 전이(이력 자동 기록).
      ```bash
      bash scripts/kim-cal.sh set <id> 작업중 "착수"
      ```
   2. 이벤트의 `요청:`을 읽고 **작업 종류**를 판별해 수행:
      - **메일 발송/답장** → gws gmail. **발송은 사용자 승인 후**(안전 규칙). 승인 못 받으면 `정보필요`.
      - **기존 문서 수정** → gws drive/docs.
      - **지식 저장** → [[kim-learn]] 증류 → [[kim-sync]] 투영.
      - **길게 조사** → 조사 후 **결과를 작업 이력/설명에 보고**.
      - **판별 불가·정보 부족** → `정보필요`.
   3. 결과로 상태 전이:
      - 성공: `set <id> 성공 "<한 일 요약/결과>"`
      - 실패: `set <id> 실패 "<이유>"` (자동 재시도 안 함)
      - 정보필요: `set <id> 정보필요 "<무슨 정보가 왜 필요한지>"` — 이게 이벤트 `필요정보:`로 사람에게 보임.
   4. **반복 작업**이면 성공 후 다음 시각 재등록:
      ```bash
      bash scripts/kim-cal.sh reenqueue <id>   # 요청/필요정보에 '평일 09:00' 등 규칙 있으면
      ```

3. **다음 작업** — due가 빌 때까지. 각 작업 독립: 하나 실패해도 다음 진행, 사용자에게 요약 보고.

## 정보필요 재개
사용자가 말로 정보를 주면(폰 또는 로컬), 해당 작업을 `set <id> 대기 "정보 보충: …"`로 되돌린다.
다음 루프에서 만기 대기로 다시 잡혀 자동 재개된다(이력에 보충 내용 남음).

## 자동화 (무인 드레인)

워커는 **gws가 인증된 곳(로컬)** 에서 돌아야 한다 — 클라우드 cron(claude.ai 루틴)은
로컬 `~/.config/gws` 자격증명이 없어 캘린더에 접근하지 못한다. 또 증류·판단에 LLM이 필요하므로
순수 cron 스크립트가 아니라 **Claude Code 에이전트 루프**여야 한다.

- **권장**: 항상 켜둔 로컬 머신의 Claude Code 세션에서
  ```
  /loop 30m /kim-work
  ```
  → 30분마다 만기 `대기` 작업을 드레인하고, 사이엔 유휴. 급하면 `5m`, 여유면 `1h`.
- "정보필요 → 대기로 되돌리면 자동 재개"는 이 루프가 돌고 있을 때 **다음 틱**에 성립한다.
- 큐가 대부분 비어 있으면 간격을 길게 — 빈 `due`는 즉시 종료라 비용이 거의 없다.

## 규칙
- 외부 부수효과(메일 발송, 문서 게시 등)는 **사용자 승인 게이트** 존중.
- `실패` 자동 재시도 없음. `성공`/`실패`도 캘린더에 남김(이력·감사).
- 모든 상태 전이는 이력에 타임스탬프와 함께. 이벤트 삭제·덮어쓰기 금지(append-only).
